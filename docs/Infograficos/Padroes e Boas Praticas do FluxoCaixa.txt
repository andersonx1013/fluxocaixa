## ✅ 3. PRINCÍPIOS SOLID — ARQUIVO POR ARQUIVO

### 📌 S — Single Responsibility Principle (SRP)

| Arquivo | Linha | Responsabilidade ÚNICA |
|---------|-------|----------------------|
| `Lancamento.cs` | 5-31 | APENAS regras de lançamento financeiro |
| `ConsolidadoDiario.cs` | 5-54 | APENAS agregação diária (débitos/créditos/saldo) |
| `LancamentoService.cs` | 11-154 | APENAS casos de uso de lançamentos |
| `ConsolidadoService.cs` | 11-167 | APENAS consolidação diária |
| `OutboxPublisherService.cs` | 11-142 | APENAS publicar eventos do Outbox |
| `RabbitMQEventConsumer.cs` | 13-154 | APENAS consumir eventos RabbitMQ |
| `RedisCacheService.cs` | 8-69 | APENAS cache Redis com fallback |
| `GlobalExceptionMiddleware.cs` | 5-47 | APENAS capturar exceções não tratadas |
| `MensagensErro.cs` | 3-12 | APENAS constantes de erro do domínio |
| `TipoLancamento.cs` | 3-7 | APENAS tipos de lançamento (enum) |

### 📌 O — Open/Closed Principle (OCP)

| Arquivo | Linha | Por que é aberto para extensão |
|---------|-------|-------------------------------|
| `ICacheService.cs` | 3-7 | Pode ter RedisCacheService, InMemoryCacheService, etc. |
| `IEventOutbox.cs` | 5-11 | Pode usar RabbitMQ, Kafka, ServiceBus, etc. |
| `ILancamentoRepository.cs` | 3-10 | Pode usar EF Core, Dapper, ADO.NET, etc. |
| `IProcessedMessageStore.cs` | 3-8 | Pode usar PostgreSQL, MongoDB, Redis, etc. |
| `IConsolidadoRepository.cs` | 3-7 | Pode usar EF Core, Dapper, etc. |
| `IUnitOfWork.cs` (ambos) | 3-6 | Pode ter diferentes implementações |

### 📌 L — Liskov Substitution Principle (LSP)

| Implementação | Interface | Pode substituir? |
|--------------|-----------|-----------------|
| `LancamentoRepository` | `ILancamentoRepository` | ✅ Sim |
| `ConsolidadoRepository` | `IConsolidadoRepository` | ✅ Sim |
| `LancamentosUnitOfWork` | `IUnitOfWork` | ✅ Sim |
| `ConsolidadoUnitOfWork` | `IUnitOfWork` | ✅ Sim |
| `OutboxEventWriter` | `IEventOutbox` | ✅ Sim |
| `ProcessedMessageStore` | `IProcessedMessageStore` | ✅ Sim |
| `RedisCacheService` | `ICacheService` | ✅ Sim |

### 📌 I — Interface Segregation Principle (ISP)

| Interface | Quantidade de Métodos | Coesa? |
|-----------|----------------------|--------|
| `ILancamentoRepository` | 5 métodos | ✅ Sim, todos de persistência |
| `IUnitOfWork` | 1 método (`SaveChangesAsync`) | ✅ Sim, mínimo necessário |
| `ICacheService` | 2 métodos (`GetOrSetAsync`, `RemoverAsync`) | ✅ Sim, só cache |
| `IProcessedMessageStore` | 2 métodos (`JaFoiProcessadaAsync`, `FinalizarProcessamentoAsync`) | ✅ Sim, só inbox |
| `IEventOutbox` | 3 métodos (um para cada tipo de evento) | ✅ Sim, só outbox |

### 📌 D — Dependency Inversion Principle (DIP)

| Arquivo | Linha | Evidência |
|---------|-------|-----------|
| `LancamentoService.cs` | 19-25 | Construtor recebe `ILancamentoRepository`, `IEventOutbox`, `IUnitOfWork` (interfaces) |
