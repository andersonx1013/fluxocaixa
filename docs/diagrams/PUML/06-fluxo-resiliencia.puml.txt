@startuml
title Fluxo Dinamico - Outbox, Queda e Recuperacao do Consolidado

actor Comerciante
participant "API Lancamentos" as Lancamentos
database "PostgreSQL\nLancamentos + Outbox" as LancDb
queue "RabbitMQ\nFila duravel" as Rabbit
participant "API Consolidado\n(indisponivel)" as Consolidado
database "PostgreSQL\nConsolidado + Inbox" as ConsDb

Comerciante -> Lancamentos: POST /api/lancamentos
Lancamentos -> LancDb: BEGIN
Lancamentos -> LancDb: INSERT lancamento
Lancamentos -> LancDb: INSERT outbox_message
Lancamentos -> LancDb: COMMIT
Lancamentos --> Comerciante: 201 Created

Lancamentos -> Rabbit: publica evento + publisher confirm
Rabbit -> Rabbit: mantem mensagem duravel\nenquanto consumidor esta offline

... API Consolidado volta a operar ...

Rabbit -> Consolidado: entrega evento (MessageId)
Consolidado -> ConsDb: verifica Inbox
Consolidado -> ConsDb: BEGIN
Consolidado -> ConsDb: UPSERT consolidado diario
Consolidado -> ConsDb: INSERT processed_message
Consolidado -> ConsDb: COMMIT
Consolidado -> Rabbit: ACK

note over Lancamentos,Rabbit
Se o broker cair, o POST continua disponivel:
o evento permanece no Outbox e e republicado.
end note

note over Rabbit,Consolidado
Se houver redelivery, o MessageId da Inbox
impede dupla contabilizacao.
end note
@enduml
