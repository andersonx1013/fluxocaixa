CONCEITOS DE SEGURANÇA

Analogia Prática

Para compreender de forma simples a nossa segurança em camadas, imagine como funciona a portaria e o acesso de um prédio comercial moderno:

🔑 AUTHN (Autenticação)

A Portaria do Prédio
É onde você prova quem você é. O porteiro olha seu documento de identidade e valida que você é realmente o Anderson Xavier. Se falhar na identificação, você é barrado com um erro 401 Unauthorized (Não Autenticado).

🎟️ JWT (JSON Web Token)

O Crachá do Visitante
Depois de provar quem você é na portaria, você recebe um crachá de acesso oficial. Esse crachá tem seus dados (claims) e um carimbo holográfico infalsificável (a assinatura offline). Qualquer segurança do prédio pode ver o crachá e validar sua integridade imediatamente sem precisar ligar para a recepção.

🚪 AUTHZ (Autorização)

As Catracas Internas
Diz respeito ao que você tem direito de fazer. Mesmo portando um crachá legítimo (autenticado), se você tentar entrar no laboratório de TI sem ter essa permissão programada no crachá, a catraca trava e bloqueia sua entrada com o erro 403 Forbidden (Acesso Proibido).

🏷️ ROLE (Papel / Perfil)

O Cargo Impresso no Crachá
Em vez de programar cada fechadura eletrônica para cada pessoa do prédio individualmente, o prédio cria perfis como "Comerciante". Se a catraca exige a Role de comerciante, só entra quem tem esse papel carimbado no crachá. Na API, usamos o atributo [Authorize(Roles = "comerciante")] nos controllers para esse bloqueio direto.

C4 DEPLOYMENT // GUIA

Escala Horizontal

Detalhamento conceitual de cada um dos 5 pilares do C4 Deployment em produção:

01. Base Real & Arquitetura Base

O que é: Estrutura fundamental desenvolvida sobre .NET 8, JWT, Health Checks, Outbox/Inbox, RabbitMQ e Redis.

Para que serve: Servir de fundação resiliente e desacoplada para suportar replicação horizontal sem necessidade de alterar o código de negócio.

Propósito: Garantir que o sistema seja nativamente preparado para ambientes em nuvem (Cloud-Native) e escalabilidade sob demanda.

02. Roteamento & Load Balancer

O que é: Camada de entrada composta por balanceador de carga HTTP (ex: NGINX / AWS ALB / Ingress Controller).

Para que serve: Distribuir a carga de requisições entre instâncias ativas e encaminhar chamadas apenas para réplicas com sondas de prontidão (Readiness Probe) válidas.

Propósito: Eliminar pontos únicos de falha de borda e evitar degradação de serviço direcionando tráfego a nós sobrecarregados.

03. Escala de Escrita (APIs & Outbox)

O que é: Multiplicação das instâncias da API de Lançamentos trabalhando com concorrência distribuída no Outbox (Claim/Lease ou Worker dedicado).

Para que serve: Absorver picos massivos de inserção de débitos e créditos mantendo locks curtos no banco de dados e evitando publicação duplicada de eventos.

Propósito: Aumentar a vazão de gravação transacional garantindo a propriedade ACID e a entrega confiável de mensagens ao RabbitMQ.

04. Escala de Leitura (APIs & Consumidores)

O que é: Multiplicação das instâncias da API de Consolidado associada a consumidores RabbitMQ operando com Single Active Consumer e nós em standby.

Para que serve: Atender altíssimo volume de consultas de saldos pelo comerciante e garantir que apenas um worker por vez consolide saldos por data para evitar race conditions.

Propósito: Desacoplar leitura de escrita e assegurar que o cálculo agregativo de saldos seja rápido, determinístico e livre de concorrência destrutiva.

05. Camada de Dados & Persistência

O que é: Clusterização de infraestrutura com réplicas e failover automático para PostgreSQL, Redis e RabbitMQ (Quorum Queues).

Para que serve: Manter réplicas de leitura para alívio do Postgres primário, distribuição de cache via Redis Sentinel/Cluster e espelhamento durável de mensagens no RabbitMQ.

Propósito: Garantir continuidade de negócios, RPO zero para dados transacionais e alta resiliência contra falhas de hardware ou nós de computação.

PADRÃO ENTERPRISE // GUIA

Itens para Evolução

Conheça os detalhes e justificativas técnicas das 3 propostas de evolução enterprise:

01. Observabilidade Distribuída

O que é: Ecossistema unificado de telemetria baseado em OpenTelemetry (OTLP) com exportação para Jaeger, Prometheus e Grafana.

Para que serve: Coletar métricas de runtime, logs estruturados e distributed traces cruzando todas as APIs e mensageria, além de monitorar o Lag de Fila no RabbitMQ.

Propósito: Fornecer visibilidade ponta a ponta em tempo real, permitindo rastrear qualquer requisição por seu TraceId e antecipar gargalos de latência.

02. Segurança & Secrets

O que é: Camada perimetral de proteção com WAF / API Gateway e gestão dinâmica de segredos via Vault / Azure Key Vault.

Para que serve: Bloquear ataques cibernéticos na borda (DDoS, SQLi, XSS), realizar offload TLS e injetar credenciais de banco e chaves JWT em runtime.

Propósito: Implementar a política de Zero Hardcoding e garantir conformidade com frameworks de segurança da informação (OWASP / LGPD).

03. Esteira CI/CD & Auditoria DLQ

O que é: Pipeline de implantação automatizada acoplada a mecanismos operacionais de tratamento de Dead Letter Queue (DLQ).

Para que serve: Executar verificações estáticas de código (SonarQube/Trivy), deploy Blue/Green e disponibilizar painel para visualização e reprocessamento (replay) de mensagens mortas.

Propósito: Assegurar entregas contínuas sem risco de downtime e fornecer governança total sobre mensagens que falharam após todas as tentativas de retry.
